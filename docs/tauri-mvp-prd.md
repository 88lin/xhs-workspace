# 小红书通用化产品 PRD

## Tauri 桌面应用 MVP

### 1. 项目背景

当前仓库本质上是一个运行在 Claude Code 中的技能插件，核心能力已经验证了这条链路可行：

- 输入小红书链接
- 复用登录态抓取帖子内容
- 提取图文和视频信息
- 对视频做转录
- 生成结构化 Markdown
- 导出到 Obsidian

现阶段的主要问题不是“能不能做”，而是“普通用户根本不会这样用”：

- 依赖 Claude Code 和 slash command
- 依赖手工导出 cookies
- 依赖本地命令行环境
- 依赖用户理解 Obsidian 目录和文件结构

因此，下一步应该把已有的工作流能力沉淀为一个独立产品，而不是继续停留在技能层。

---

### 2. 产品目标

#### 2.1 目标

做一个让普通用户几乎不需要技术背景就能使用的桌面应用，完成以下事情：

- 一键导入小红书帖子
- 自动抓取图文或视频内容
- 自动生成摘要、标签和结构化笔记
- 支持本地资料库管理
- 一键导出到 Obsidian 或 Markdown 文件夹

#### 2.2 非目标

MVP 阶段不做以下内容：

- 网页 SaaS
- 手机 App
- 小红书全站搜索或爬取
- 自动同步用户全部收藏夹
- 团队协作
- 云端知识库
- 复杂商业化订阅系统

---

### 3. 产品定位

一句话定位：

> 把小红书内容一键沉淀为可搜索、可转录、可导出的本地资料库。

第一阶段不追求“大而全”，只服务以下高价值用户：

- 知识管理用户：想把收藏内容沉淀到 Obsidian
- 研究型用户：需要批量整理教程、方法论、竞品内容
- 创作者：需要把分散收藏转成结构化素材

---

### 4. 为什么选 Tauri

相对网页或浏览器插件，Tauri 更适合当前项目的真实能力边界。

#### 4.1 优势

- 允许本地文件系统读写，适合导出 Markdown 和 Obsidian
- 适合承载登录态和受控抓取流程
- 安装包和运行资源通常比 Electron 更轻
- 可以调用 Rust 原生命令，适合任务队列、下载、解析、缓存管理
- 可以通过 sidecar 方式打包外部二进制，适合后续引入 `ffmpeg` 或本地转录能力

#### 4.2 约束

- 比纯前端页面更偏工程化，初期开发门槛更高
- 本地视频转录如果走 sidecar，会增加打包和跨平台复杂度
- 桌面端登录态管理必须从第一天就做稳，否则后续问题会集中爆发

#### 4.3 产品结论

MVP 采用：

- `Tauri 2`
- `React + TypeScript + Vite`
- Rust 负责任务编排、抓取、缓存、导出
- 前端负责页面交互、状态展示、资料浏览

### 5. 核心用户与使用场景

#### 5.1 用户画像 A：Obsidian 用户

特征：

- 已经有自己的知识库
- 经常收藏小红书教程、方法论、经验帖
- 不想手动复制粘贴整理

核心诉求：

- 一键保存
- 结构清晰
- 能导出到既有知识库

#### 5.2 用户画像 B：研究型整理用户

特征：

- 会同时收集多条同主题帖子
- 对批量导入、搜索、标签管理敏感

核心诉求：

- 批量处理
- 可搜索
- 可比较

#### 5.3 用户画像 C：内容创作者

特征：

- 收藏灵感多，但后续难复用
- 希望快速得到摘要和可编辑素材

核心诉求：

- 提炼重点
- 快速回看
- 导出方便

### 6. MVP 核心承诺

用户完成首次登录后，只需要做两件事：

1. 粘贴一个或多个小红书链接
2. 点击导入

系统自动完成：

- 登录态复用
- 页面抓取
- 内容解析
- 媒体下载
- 视频转录
- 摘要整理
- 本地保存
- 导出到指定目录

### 7. MVP 功能范围

#### 7.1 P0 必须做

| 模块 | 功能 | 说明 |

| 模块 | 功能 | 说明 |

| 任务 | 队列状态与进度 | 看到处理中、成功、失败 |
| 详情 | 原文与摘要双栏查看 | 提高复查效率 |

#### 7.3 P2 可以延后

- 主题分析
- 多条笔记对比
- 自动聚类
- Notion 导出
- 飞书导出
- 自动监控剪贴板
- 浏览器分享入口
- 云同步

### 8. 用户主流程

#### 8.1 首次使用流程

1. 用户打开应用
2. 进入欢迎页
3. 进入登录页
4. 在内嵌登录窗口中完成小红书登录
5. 登录成功后进入首页
6. 设置默认导出目录
7. 开始导入第一条链接

#### 8.2 单条导入流程

1. 粘贴链接
2. 点击导入
3. 系统校验链接格式
4. 使用登录态请求页面
5. 解析帖子数据
6. 如为视频，下载并转录
7. 生成摘要和标签
8. 存入本地资料库
9. 显示详情预览
10. 用户可选择导出

#### 8.3 批量导入流程

1. 用户进入批量导入页
2. 粘贴多条链接
3. 系统去重
4. 逐条进入任务队列
5. 用户在任务页查看进度
6. 失败项支持重试

### 11. 功能详细需求

#### 11.1 登录与会话管理

要求：

- 用户在应用内完成小红书登录
- 会话默认保存在本地
- 会话失效时要能明确提示并引导重新登录
- 不要求用户手工导出 cookie 文件

验收标准：

- 用户首次登录后，无需再使用 DevTools
- 登录失效时，导入流程不会静默失败

#### 11.2 帖子导入

要求：

- 支持标准帖子链接粘贴
- 自动识别单条和多条链接
- 自动去重
- 导入失败要有明确原因

验收标准：

- 图文帖子导入成功率稳定
- 批量导入中单条失败不影响其他任务继续

#### 11.3 内容解析

要求：

- 提取帖子标题、正文、图片、视频、作者、发布时间、互动数据
- 保存原始结构化数据，便于后续重新生成摘要

验收标准：

- 至少支持图文帖和视频帖两种主要类型

#### 11.4 转录

MVP 建议：

- 默认优先云端转录，降低安装和打包复杂度
- 预留本地转录模式接口，后续可接 sidecar

验收标准：

- 视频帖子转录后可在详情页查看文本
- 转录失败时不影响帖子主体入库

#### 11.5 AI 摘要与结构化整理

要求：

- 生成一句话摘要
- 生成 3 到 5 个标签
- 生成结构化内容区块

验收标准：

- 输出可直接作为资料卡片阅读
- 用户可以重新生成，不需要重复抓取帖子

#### 11.6 导出

要求：

- 一键导出为 Markdown
- 支持导出到指定 Obsidian 目录
- 导出内容包含标题、摘要、元信息、正文整理、转录文本

验收标准：

- 用户能在外部目录直接看到生成文件
- 重复导出有清晰策略：覆盖 / 跳过 / 另存

### 13. 技术架构建议

#### 13.1 前端

- `React + TypeScript + Vite`
- 路由：页面壳、资料库、详情、设置
- 状态管理：轻量 store 即可，优先简单

#### 13.2 Tauri/Rust 层

负责：

- 登录窗口控制
- Cookie / 会话读取
- 网络请求
- 下载任务
- 本地 SQLite 存储
- 文件导出
- 缓存清理

#### 13.3 存储层

- `SQLite`：结构化数据
- `AppData/cache`：图片、视频、临时文件
- 用户设定的导出目录：最终 Markdown 文件

#### 13.4 转录层

MVP 推荐：

- 接云端 ASR 服务
- 本地转录作为后续能力

后续可扩展：

- `ffmpeg` sidecar
- 本地 ASR sidecar

#### 13.5 AI 层

建议拆成可替换接口：

- `summarizePost(rawPost)`
- `generateTags(rawPost, transcript)`
- `buildMarkdown(summary, rawPost, transcript)`

这样后面切模型或换供应商不会动页面层。

### 14. Tauri 实现建议

#### 14.1 推荐技术栈

- 桌面壳：`Tauri 2`
- UI：`React + TypeScript`
- 存储：`SQLite`
- 网络：Rust 请求层
- 导出：Rust 文件写入
- 队列：Rust 后台任务

#### 14.2 关键技术决策

1. 登录态不要再走“手动导出 cookies”
2. 解析结果要保存 `raw_payload`，避免每次重新抓取
3. 转录失败不能阻断整个帖子入库
4. AI 摘要与抓取流程解耦，便于重算
5. 所有失败项都要有明确错误原因

#### 14.3 目录建议

    tasks/
    settings/
src-tauri/
  src/
    commands/
    services/
    session/
    storage/
    exporter/
    tasks/
    parser/

### 16. 风险与应对

#### 16.1 平台结构变动

风险：

- 页面字段变化导致解析失败

应对：

- 解析逻辑模块化
- 保存原始数据
- 增加失败日志和快速修复点

#### 16.2 登录态失效

风险：

- 用户无法稳定导入

应对：

- 明确登录状态检测
- 失效时直接引导重新登录

#### 16.3 转录成本与复杂度

风险：

- 本地打包太重，跨平台成本高

应对：

- MVP 先上云端转录
- 本地模式延后

#### 16.4 用户预期过高

风险：

- 用户误以为这是全自动抓取器或全站采集器

应对：

- 产品文案强调“处理你主动导入的链接”

### 18. 第一版最终定义

如果只保留一句最重要的产品定义：

> 这是一个本地优先的 Tauri 桌面工具，帮用户把主动提交的小红书链接转成可整理、可搜索、可导出的知识资料。
